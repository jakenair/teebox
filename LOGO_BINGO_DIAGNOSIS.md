# Logo Bingo cross-platform diagnosis

**Symptom:** Logo Bingo shows the same daily puzzle with *different* logos
on web vs iOS. Both should show the identical 9 logos in the same order
for any given day.

## What the codebase actually does today

### (a) Where logo assets live

| Surface | Path | Purpose |
|---|---|---|
| Source PNGs | `/assets/logos/{course-id}.png` (118 files) | Canonical bytes, served from the same web origin |
| Logo manifest | `/assets/logos/manifest.js` (`LOGOS_AVAILABLE` Set, 117 entries) | Auto-generated by `scripts/generate-logos-manifest.mjs` from `assets/logos/*.png`. Filters the COURSES pool down to courses that actually have a PNG. |
| Course pool | `/bingo-courses.js` (`COURSES` array, ~140 entries) | Course metadata + helper `courseLogoUrl(course)` → `/assets/logos/{id}.png` |
| iOS Xcode asset catalog | `ios/App/App/Assets.xcassets/` — only `AppIcon.appiconset` + `Splash.imageset` | **No logo images are bundled in the Xcode project itself.** |
| iOS bundled web payload | `ios/App/App/public/` (created by `npm run build:web`) | `rsync -a dist/ ios/App/App/public/` — this is where iOS gets its copy of `bingo-courses.js`, `manifest.js`, and `/assets/logos/*.png` |

### (b) How the daily puzzle is generated

Single client-side function, `dailySeed(dateStr)` in `index.html` (lines
15303–15324):

```js
const rngOrder = mulberry32(hashStr('teebox-bingo-canon-v3'));
const pool = COURSES.filter(c => LOGOS_AVAILABLE.has(c.id));
for (let i = pool.length - 1; i > 0; i--) {  // shuffle
  const j = Math.floor(rngOrder() * (i + 1));
  [pool[i], pool[j]] = [pool[j], pool[i]];
}
const today = new Date(dateStr + 'T00:00:00Z');
const epoch = new Date('2026-01-01T00:00:00Z');
const daysSince = Math.max(0, Math.floor((today - epoch) / 86400000));
const windowCount = Math.max(1, Math.floor(pool.length / 9));
const start = (daysSince % windowCount) * 9;
return pool.slice(start, start + 9);
```

The output is fully determined by:
1. The seed string literal `'teebox-bingo-canon-v3'`
2. The exact contents of `COURSES` (order matters — shuffle is stable but
   inserts/removes change the result)
3. The exact contents of `LOGOS_AVAILABLE` (filter step)
4. The epoch `2026-01-01T00:00:00Z`

No server is consulted. There is no `dailyPuzzles` Firestore doc. The
prior `generateDailyBingoPuzzle` Cloud Function was *deleted* on
2026-05-12 (see `BINGO_CLEANUP.md`) because — quoting that doc — *"Even
if a reader appeared, it would disagree with the client: the function
used Math.random() while the client uses a mulberry32 PRNG seeded by
date hash."*

### (c) Where logos render

Two render sites, both in `index.html`:

1. **Grid renderer** (line ~15876): for each of the 9 cells, calls
   `courseLogoUrl(course)` and sets an `<img src>` to
   `/assets/logos/{id}.png`. Probes the image with `new Image()`; if the
   probe loads, swaps it in over the emoji placeholder.
2. **Share-card generator** (line ~16121): same `courseLogoUrl()` call.

Both web and iOS render through this same JS — iOS has no native bingo
screen. It is a Capacitor WKWebView loading the same `index.html`.

## Specific answers

1. **Is iOS bundling logo images locally in the Xcode project?**
   No native bundling. The PNGs are copied into `ios/App/App/public/assets/logos/`
   by `npm run build:web` and served from the WebView's own origin. The
   Xcode `Assets.xcassets` contains only the app icon and splash — no
   course logos.

2. **Is web fetching from the same URL/CDN that iOS uses, or different?**
   Web fetches from `https://teeboxmarket.com/assets/logos/{id}.png`
   (GitHub Pages origin). iOS fetches from the WebView's local origin
   (`capacitor://localhost/assets/logos/{id}.png` or the equivalent),
   reading the file that `npm run build:web` copied into
   `ios/App/App/public/`. **Different sources, even when the bytes
   match.** If the App Store IPA was built from an older `dist/` than
   what's live on teeboxmarket.com, the PNG bytes and the
   `bingo-courses.js` / `manifest.js` shipping with it are stale.

3. **Is the daily puzzle generated once server-side, or per-client?**
   **Per-client.** Each client runs `dailySeed()` independently. No
   server document is consulted.

4. **Are logo metadata stored in one place or duplicated?**
   The *source* is one place (`/bingo-courses.js`, `/assets/logos/`).
   The *deployed copy* is duplicated: one lives in the GitHub Pages
   web origin, one is rsync'd into the iOS bundle on each
   `npm run cap:sync`. They are identical at build time but drift the
   moment web ships without an iOS resync.

## Root cause classification

**Primary: ROOT CAUSE C — each client runs its own puzzle selection
logic.** Even though both clients run the same JS function, the function
is sensitive to the snapshot of `COURSES` + `LOGOS_AVAILABLE` shipped
with the bundle. There is no server-side coordination.

**Contributing: ROOT CAUSE A — assets duplicated across platforms.**
The PNGs and metadata files exist as two copies (web origin + iOS
bundle) instead of one canonical CDN URL pair. Any drift in either
direction (an `assets/logos/` update deployed to web but the iPhone
hasn't been resync'd; or a `bingo-courses.js` update with a new course
added between web-deploy and IPA-build) produces a different `pool`,
which produces a different daily window, which produces *different 9
logos* on the same UTC date.

The two combine: the deterministic shuffle is *only* deterministic per
snapshot of inputs. When the inputs differ between platforms, the
shuffle outputs differ, and there is no server arbitration to detect
or correct the divergence.

## Fix outline (executed in subsequent commits)

1. Reinstate a single source of truth: a scheduled Cloud Function
   `generateDailyBingoPuzzle` writes `dailyPuzzles/{YYYY-MM-DD}` at UTC
   midnight with `{ date, courses: [{id, shortName, logoUrl}], ... }`.
   The function uses the **same deterministic mulberry32 + canonical
   seed** the client uses today, executed once server-side. This
   guarantees both clients pull identical data.
2. Refactor the client to fetch `dailyPuzzles/{today}` on bingo open.
   Keep `dailySeed()` as an offline fallback only, with
   `console.warn('[bingo] offline fallback — server puzzle unreachable')`
   so the founder can spot offline-mode hits in the logs the next time
   web-vs-iOS divergence is reported.
3. The Firestore doc carries the canonical CDN URL per logo
   (`https://teeboxmarket.com/assets/logos/{id}.png`) so even if the
   iOS bundle's copy is stale, the rendered `<img src>` points at the
   live web origin.
4. Add monitoring: a daily Cloud Function compares what iOS would show
   for today against what web would show for today (both pull
   `dailyPuzzles/{today}` — comparison should always be identical; alert
   if not). Plus an audit script that hashes every logo's CDN URL
   against an expected manifest so silent uploads can't replace the
   image bytes under an existing ID.
5. CI script that fails if iOS code tries to bundle logo images in
   `Assets.xcassets/` or if client-side puzzle generation paths
   reappear without the explicit `offline-fallback` marker.
