# Logo Bingo — Caching Efficiency Audit

**Repo:** `/Users/jakenair/Desktop/teebox`
**Branch:** `main` @ `b208c2c`
**Scope:** read-only audit. No code modified, no commits.
**Economics under audit:** same 9 logos serve every player globally for 24h → cost should be near-zero. Any per-user fetch, per-tap write, or uncached image request is waste.

---

## 1. Architecture overview

Logo Bingo is **fully client-side puzzle generation** on top of static assets, with **opt-in server writes only at game-completion** for the leaderboard.

| Layer | Where it lives | Delivery |
|---|---|---|
| Course pool (90+ entries) | `/bingo-courses.js` (`COURSES` export) | Static JS, served from origin (currently GitHub Pages), precached by `sw.js` |
| Available-logo manifest | `/assets/logos/manifest.js` (`LOGOS_AVAILABLE`) | Static JS, auto-generated by `scripts/generate-logos-manifest.mjs` |
| Daily 9-tile selection | `dailySeed(TODAY)` in `index.html:14528-14543` | Pure client function: `mulberry32(hashStr('teebox-bingo-canon-v3'))` shuffles the filtered pool once, slices `[start, start+9)` where `start = (daysSince(2026-01-01) % windowCount) * 9` |
| "TODAY" date key | `todayUTC()` in `index.html:14478-14485` | `Intl.DateTimeFormat('en-CA', { timeZone: 'America/New_York' })` — note: function is named "UTC" but actually uses **America/New_York** wall-clock. Same global flip moment, fine. |
| Logo images | `/assets/logos/{course.id}.png` (82 files, ~1.07 MB total, avg ~13 KB each, max ~17 KB) | Same-origin static, GitHub Pages |
| Player progress | `localStorage['teebox.bingo.{YYYY-MM-DD}']` and `localStorage['teebox.bingo.streak']` | Local-only. Every tap writes JSON via `persist()` |
| Leaderboard | `gameScores/{date}_{uid}` (Firestore) | **One** `setDoc(..., {merge:true})` per puzzle completion (`writeLeaderboardScore`, `index.html:15305-15327`). Anonymous users skip silently. |
| Server-generated puzzle doc | `dailyGames/{YYYY-MM-DD}` (Firestore) | `generateDailyBingoPuzzle` Cloud Function fires daily at 00:00 UTC (`functions/index.js:2552-2579`). **Currently unread by client** — vestigial. |
| Service worker | `sw.js` v `teebox-v1-2026-05-12-r86` | Precaches `/bingo-courses.js` (shell, line 8). Bypasses `/assets/logos/*` entirely (lines 77-82) — comment says "we don't want users pinned to a stale copy". |

**One-sentence summary:** the daily puzzle is **derived locally** on every client from a date-seeded deterministic shuffle of the static `bingo-courses.js` pool — no Firestore reads to play, one Firestore write per completed game (signed-in only), 82 PNG logos served as plain origin static assets.

## 2. Property-by-property verdict

### Property 1 — Puzzle generated once and stored as a single immutable doc

| | |
|---|---|
| **Actual** | Two parallel mechanisms exist: (a) **client-side deterministic derivation** at `index.html:14528-14543` (this is the one actually used), and (b) a **server-side scheduled write** to `dailyGames/{YYYY-MM-DD}` in `functions/index.js:2552-2579` — but **no client code reads that doc** (grep `dailyGames` only matches the function itself). The server function uses `Math.random()` (non-deterministic), while the client derives from a date-seeded `mulberry32`. |
| **Ideal** | One source of truth. Either: client-derived only (and delete the cron + Firestore doc + the rule at `firestore.rules:829-832`), or server-generated + client-fetched (with cache headers). |
| **Severity** | **MED** (correctness) / **LOW** (cost). Doesn't cost real money — the function runs once/day and is idempotent (`functions/index.js:2630-2634`). But it's misleading: any future engineer who reads only `firestore.rules:825-828` ("every player on a given day gets the same 9 courses … created by a scheduled Cloud Function") will be confused, because the client never reads it. Also: the server's `Math.random()` pick and the client's seeded shuffle produce **different** 9-tile sets, so if someone ever wires the client to read `dailyGames`, the game changes shape silently. |
| **Fix** | Delete the dead server path. **Diff sketch:** <br>`functions/index.js:2499-2579` — remove the entire `generateDailyBingoPuzzle` block and the `BINGO_COURSE_POOL` constant. <br>`firestore.rules:824-832` — remove the `match /dailyGames/{date}` block. <br>`firestore.indexes.json` — check for any index on `dailyGames` (none referenced in code, but verify). |
| **Cost today** | ~0. One scheduled invocation/day + one 1-doc write/day = $0.0000018/day in writes + free-tier compute. |

### Property 3 — CDN cache with 24h max-age + stale-while-revalidate

| | |
|---|---|
| **Actual** | Hosting is **GitHub Pages** (per `CNAME`, `docs/MIGRATION_FIREBASE_HOSTING.md`, `userMemory`). GH Pages applies its own defaults — typically `Cache-Control: max-age=600` for HTML and ~10 min CDN edge cache. The `firebase.json:36-134` cache headers (including `assets/**: max-age=31536000, immutable`) are **dead code** until the Firebase Hosting cutover happens. <br><br>`sw.js` handles `/bingo-courses.js` with **stale-while-revalidate** (line 108-126: same-origin non-HTML branch). That's effectively unlimited browser-side cache + background refresh — ideal for a static, daily-rotating dataset. <br><br>The puzzle has **no filename hash** (`/bingo-courses.js`, not `/bingo-courses.{hash}.js`). When you add a course, every cached browser keeps the old file until SW's background revalidate runs *and* the SW's `CACHE_VERSION` is bumped (line 3, currently `r86`). Forgetting to bump the cache version is the documented bug pattern (`sw.js:60-62`). |
| **Ideal** | (a) `bingo-courses.js` either content-hashed in filename or served with `Cache-Control: public, max-age=86400, stale-while-revalidate=604800`. (b) Migrate to Firebase Hosting so `firebase.json` headers go live, then the `**/*.@(js|css|svg)` rule (`firebase.json:111-114`) applies `max-age=2592000, s-maxage=31536000`. |
| **Severity** | **LOW** — the SW already gives us stale-while-revalidate for free for repeat visitors. **MED** for first-visit users behind ISP caches that respect GH Pages's short TTL. |
| **Fix** | Two options, pick one: <br>**(A)** Finish the Firebase Hosting migration (`docs/MIGRATION_FIREBASE_HOSTING.md`); headers in `firebase.json:36-134` activate automatically. <br>**(B)** Add a content-hashed filename. **Diff sketch** in `package.json:7` (build script): inject a hash, e.g. `cp bingo-courses.js dist/bingo-courses.$(md5 -q bingo-courses.js | cut -c1-8).js` and rewrite the import in `index.html:14469`. Too invasive for the marginal win — stick with (A). |
| **Cost today** | $0 (GH Pages bandwidth is free). Latency cost: ~50-200 ms on first visit for the 87 KB file. |

### Property 4 — Logo images served from CDN with long max-age + content-hashed URLs

| | |
|---|---|
| **Actual** | 82 PNG files in `/assets/logos/*.png`, total 1.07 MB, average 13 KB, max ~17 KB (range 2.8-17.8 KB). Filenames are **plain** (`augusta-national.png`, not `augusta-national.v3.png`). <br><br>`sw.js:77-82` **explicitly bypasses** the service worker for `/assets/logos/*` — comment: "we don't want users pinned to a stale copy". This means the browser HTTP cache applies. GitHub Pages sends a `Cache-Control: max-age=600` for images by default. <br><br>`firebase.json:104-109` would serve `**/*.@(png|jpg|...)` with `max-age=31536000, immutable` — but again, that file isn't active yet. <br><br>Render path (`index.html:14761-14772`): on every `render()` call the code probes each logo via `new Image()` inside `forEach`. That probe relies on browser HTTP cache; with GH Pages' 600s default the 9 images re-validate every 10 min per session. <br><br>**Positive:** the SW bypass means `LOGOS_AVAILABLE` (`assets/logos/manifest.js`) filters logos to only those that ship — no 404 probes. <br><br>**Format:** PNG, not WebP/SVG. At ~13 KB/logo these are already small. SVG would be marginal at best (some of these are photo crops, not flat marks). |
| **Ideal** | PNGs renamed to include a content hash so `Cache-Control: max-age=31536000, immutable` is safe; SW caches them with cache-first strategy. With 100k DAU × ~117 KB (9 × 13 KB) = ~11 GB/day on first visit, free thereafter. |
| **Severity** | **MED** while on GitHub Pages (active re-validation every 10 min costs network round-trips, not money). **LOW** after Firebase Hosting migration. **HIGH** if a future agent ever moves logos to Firebase Storage without setting cache metadata — at 100k DAU × 1.07 MB total = ~100 GB/day = ~$12/day egress. |
| **Fix** | Two steps: <br>**(A)** Complete the Firebase Hosting migration. The `**/*.@(png|...)` immutable header in `firebase.json:104-109` then applies. <br>**(B)** When logos are "final" (per the comment in `sw.js:77-79`), remove the SW bypass at `sw.js:80-82` so the SW's same-origin stale-while-revalidate (lines 108-126) picks them up too — gives offline play and survives slow networks. <br>**(C, optional)** Content-hash filenames. The build script (`package.json:7`) could rename `assets/logos/{id}.png` → `assets/logos/{id}.{hash}.png` and regenerate `LOGOS_AVAILABLE` accordingly. More work than worth at our scale. |
| **Cost today** | $0 (GH Pages). Future at 100k DAU on Firebase Hosting: <1c/day with proper headers, ~$12/day without. |

### Property 5 — User progress saves locally first, syncs to server

| | |
|---|---|
| **Actual** | **Excellent.** Every tap path (`submitGuess` `index.html:15086-15111`, `skipGuess` `index.html:15113-15130`) calls `persist()` (line 14671-14678), which synchronously writes the full 9-cell state to `localStorage`. **No Firestore write per tap.** <br><br>Firestore write happens **once** in `writeLeaderboardScore()` (`index.html:15305-15327`) called from `maybeFinish()` (line 15302), which only fires when `isComplete()` returns true — i.e., when all 9 tiles are resolved. <br><br>Anonymous / guest users skip the write entirely (`if (!u \|\| !u.uid) return;` at line 15308). Errors are swallowed (line 15322-15326). <br><br>**No offline queue** — if the user finishes offline and never re-opens, their score never reaches Firestore. With single-shot scoring this is acceptable; the local score is the source of truth for streaks and UI. <br><br>**Streak math is client-side** (`maybeFinish` line 15262-15275), but `firestore.rules:838-840` claims "Cloud Functions perform the authoritative streak math server-side" — search of `functions/index.js` shows **no** such function exists. The client passes its own `streak` value into the leaderboard write, and the rules allow it through (`firestore.rules:858-860`). A motivated cheater could set `streak: 9999` in devtools and the rule accepts it. <br><br>**Cell flip optimization:** `render()` (line 14723-14840) does a full `innerHTML = ''` rebuild on every state change, then re-probes all 9 logo images via `new Image()`. With browser cache this is free, but it's wasteful — each tap triggers 9 probe requests that may all be 304s. |
| **Ideal** | Identical except: (a) document the lack of offline-replay-queue if it's intentional; (b) either implement the server-side streak validation the rules comment promises, or update the comment and lock down `streak` field updates more strictly. |
| **Severity** | **LOW** (cost). **MED** (integrity — streak is client-trusted despite rules-comment claiming otherwise). |
| **Fix** | <br>**Cost (LOW):** none — already optimal. <br>**Integrity (MED):** <br>Option A — add `validateBingoScore` callable in `functions/index.js`; reject leaderboard writes that don't go through it. <br>Option B — write a Firestore `onCreate` trigger on `gameScores/{scoreId}` that recomputes streak from the prior day's score doc, and overwrites the client-supplied `streak`. <br>Option C — accept current state, fix the misleading comment in `firestore.rules:838-840` & `:862-863`. <br>**Render optimization (LOW):** in `render()` (`index.html:14761-14772`), cache the resolved logo `<img>` per cell index and skip the `new Image()` probe if the cell's state hasn't changed. Saves ~9 conditional GETs per tap. <br>**Diff sketch** (render optimization): <br>```js<br>// At top of render(): track last-rendered course ids per cell<br>const renderedLogos = render.__rendered ||= {};<br>// Inside the querySelectorAll loop, skip the probe if<br>// renderedLogos[i] === course.id && el.querySelector('img')<br>``` |
| **Cost today** | 1 write per completed game per signed-in user. At 100k DAU × 0.5 completion rate × ~0.5 signed-in rate = 25k writes/day = $0.045/day. Negligible. |

## 3. Cost model — current vs ideal

Napkin math at hypothetical 100k DAU (we're nowhere near this; included as a sensitivity check).

| Cost line | Current (GH Pages) | Ideal | Annual |
|---|---|---|---|
| Puzzle delivery (P1, P2) | $0 — client-derived from static JS bundle | $0 (same) | $0 |
| Puzzle data shipped per user | 87 KB in `bingo-courses.js`, cached after first load by SW | Same | 0 net once cached |
| Logo image bandwidth | 1.07 MB / new visitor (GH Pages free egress) | 1.07 MB / new visitor (Firebase Hosting: 100k × 1.07 MB / day worst-case = 100 GB/day × $0.15/GB = **$15/day** if cache misses; with `max-age=31536000 immutable` after firsthand visit: ~$1/day for steady-state new visitors) | $5-30/yr |
| Server puzzle generation (P1 dead path) | 1 scheduled run/day + 1 write/day | Delete it | -$0 (negligible savings) |
| Per-tap Firestore writes | 0 (local only) | 0 (same) | $0 |
| Per-completion Firestore writes | ~1 write × 25k completed/day = 25k writes/day × $1.80/M = **$0.045/day** | Same | ~$16/yr |
| Leaderboard reads | Each open of bingo screen → `getDocs(... in [7 dates], limit 50)` = 1 query, returns up to 50 docs = 50 reads per screen open. At 100k DAU × 1.2 opens avg = 6M reads/day × $0.06/100k = **$3.60/day** | Cache the top-50 in a Firestore doc updated by a scheduled function; clients read 1 doc | $1300/yr (current) vs $30/yr (ideal) |

**Note:** the **leaderboard query is the actual hot path** for cost, not the puzzle delivery. It's outside the strict scope of this audit (audit focuses on puzzle delivery + image cache + progress), but it's the biggest line item if usage grows. Flagging here.

## 4. Implementation order (highest ROI per LOC first)

1. **(20 LOC) Delete the dead `generateDailyBingoPuzzle` Cloud Function** + its Firestore rule + `BINGO_COURSE_POOL` constant.
   - **Files:** `functions/index.js:2499-2579`, `firestore.rules:824-832`.
   - **Win:** removes ~75 LOC of confusing dead code; reduces "what is the source of truth?" risk for future engineers.
   - **Risk:** none — confirmed by grep that no client code reads `dailyGames`.

2. **(5 LOC + ops) Finish the Firebase Hosting migration** per `docs/MIGRATION_FIREBASE_HOSTING.md`.
   - **Win:** activates `firebase.json:104-114` headers — bingo logos become `max-age=31536000 immutable`, `bingo-courses.js` becomes `max-age=2592000`. Closes the "is anything actually cached at the edge?" question.
   - **Risk:** DNS cutover ops work, not code.

3. **(~12 LOC) Render-optimization in `index.html` `render()`** — skip re-probing logos when the cell's course id hasn't changed.
   - **Win:** stops ~9 conditional GETs per tap. Cosmetic for cost but real for perceived snappiness on slow networks.

4. **(~30 LOC) Server-side streak validation** to match the comment in `firestore.rules:838-840`.
   - **Win:** integrity, not cost. The current comment overstates what's enforced.

5. **(~80 LOC) Cached top-50 leaderboard doc** updated by scheduled function (out-of-scope for this audit but the biggest cost lever).

6. **(low priority) Content-hash logo filenames** + remove `sw.js` bypass at line 80-82.
   - Only worth doing once the logos are "final" per the existing comment.

## 5. Open questions

1. **Is the `generateDailyBingoPuzzle` Cloud Function intentionally vestigial** (kept for a future server-validated puzzle path), or was it superseded by client-side derivation and never cleaned up? The function comment (`functions/index.js:2499-2506`) and the rule comment (`firestore.rules:824-828`) both read as if the client reads from it; the code clearly doesn't.

2. **Is the Firebase Hosting migration still planned?** `docs/MIGRATION_FIREBASE_HOSTING.md` is detailed but `CNAME` still points at `teeboxmarket.com` and `userMemory` says GH Pages is live. Most caching recommendations here resolve once the migration runs.

3. **Is anonymous-user score recovery a goal?** Currently if a guest plays, finishes, then signs up later, their score is local-only. The leaderboard write at `index.html:15314` keys on `${TODAY}_${u.uid}` and skips if no `u.uid`. Should we replay on sign-in?

4. **Is the client-supplied `streak` value trusted by design?** The Firestore rule (`firestore.rules:858-860`) allows it through with bounds checks; the comment claims server-authoritative math; the server function doesn't exist. Pick one.

5. **What's the expected DAU range?** All cost numbers above scale linearly. At our current ~hundreds of DAU, every line in the table is sub-cent. At 100k DAU, the leaderboard query dominates everything else by 70-100x. The audit prompt asks about puzzle delivery economics — for which the answer is "the architecture is already at $0/day, no fixes needed for cost". The integrity and dead-code findings stand regardless.
